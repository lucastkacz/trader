# Simulation Architecture

The simulation lab should be built as an offline verification system around the
existing trading modules. Its job is to provide controlled inputs, replay them
through public interfaces, and assert safety outcomes.

The initial implementation can create only a subset of this tree. The docs
define the target shape so the package does not grow by accident.

## Dependency Direction

Allowed:

```text
simulation -> src
tests      -> simulation
tests      -> src
```

Forbidden:

```text
src -> simulation
```

Simulation helpers must not become hidden production dependencies.

## Configuration Boundary

Scenario files should be YAML because the simulation lab is meant to be read,
edited, and generated by both humans and AI agents.

Required flow:

```text
YAML scenario file
-> schema parser
-> typed scenario config
-> generators/replay/assertions
```

Rules:

- YAML is declarative only.
- YAML contains no formulas or executable expressions.
- Raw YAML dictionaries must not leak below `simulation/schema`.
- Lower layers receive typed objects.
- Validation errors should name the scenario id, field path, and expected type.
- Defaults are allowed only when modeled explicitly in the typed schema.

This mirrors the production config boundary in `src/engine/trader/config`.

## Core Concepts

### Scenario

A scenario is a deterministic recipe for:

- Market path generation.
- Data-quality faults.
- Pair artifacts.
- Initial runtime state.
- Runtime config.
- Replay steps.
- Expected outcomes.
- Safety invariants.

### Phase

A phase is a bounded interval of bars that modifies the scenario:

- Normal cointegrated behavior.
- Spread shock.
- Slow reversion.
- Correlation breakdown.
- Missing data.
- Recovery.

Phases should be composable and order-aware.

### Replay

A replay executes the scenario through public runtime interfaces. The first
target should be `execute_tick` with an injected market-data provider. Later
targets may include command processing, reporting, reconciliation, and full
runtime loop drills.

### Assertion

An assertion verifies a business or safety invariant. Assertions should report
what failed, where it failed, and which seed generated the failure.

## Public Interfaces To Prefer

Initial simulation should use:

- `src.engine.trader.runtime.tick.execute_tick`
- `src.engine.trader.runtime.pair_queue`
- `src.engine.trader.runtime.pair_validity`
- `src.engine.trader.runtime.artifacts`
- `src.engine.trader.state.manager.TradeStateManager`
- `src.engine.trader.reporting.generate_report`
- `src.engine.trader.reconciliation.run_read_only_audit`

Avoid private helpers when a public module interface exists.

## Required Runtime Seam

The first implementation likely needs a small market-data seam in `execute_tick`.

Current behavior:

```text
execute_tick -> _fetch_pair_candles -> fetch_recent_candles
```

Target behavior:

```text
execute_tick(..., market_data_provider=None)
```

Production default:

```text
market_data_provider = real exchange/data fetcher
```

Simulation default:

```text
market_data_provider = synthetic OHLCV provider
```

The seam must preserve production behavior and tests must prove that state-only
simulation does not reach network or exchange mutation paths.

## Configuration Rules

Simulation config should use typed objects below the scenario definition layer.

Required explicit fields:

- Scenario id.
- Seed.
- Timeframe.
- Exchange id.
- Symbols.
- Bar count.
- Start timestamp.
- Output directory.
- Execution mode.

No hidden operational defaults should decide paths, exchange, timeframe, or
state DB location.

## Library Strategy

The simulation lab should use high-quality base libraries for math and testing,
while keeping the scenario, replay, and invariant engine custom to this
platform.

Preferred base libraries:

- `numpy` for deterministic vectorized path generation.
- `pandas` for OHLCV frames and time indexes.
- `scipy` for distributions, calibration helpers, and statistical utilities.
- `statsmodels` for cointegration and time-series diagnostics where useful.
- `hypothesis` for property-based and fuzz tests after deterministic examples
  are stable.
- `numba` for hot path kernels only when profiling proves the need.

Avoid using heavy external backtesting engines as the core replay system. They
can inspire ideas such as vectorized data handling, event-driven data feeds, and
broker simulation, but they do not understand this platform's artifacts,
pair-validity semantics, dynamic queue, natural-exit policy, state-only
execution, command processing, or reconciliation rules.

The custom part of the lab is the professional verification layer:

- Scenario DSL.
- Replay harness.
- Market-data provider seam.
- Artifact simulator.
- Pair-validity controls.
- Queue/risk/reconciliation assertions.
- Safety invariant library.
- Result artifacts.

External libraries should support this layer, not replace it.

## Output Rules

Every replay should be able to emit:

- Scenario metadata.
- Seed.
- Git commit when available.
- Runtime config summary.
- Input artifact paths.
- Output state DB path.
- Assertion results.
- Failure summary.
- Generated market-data parquet paths.
- Generated pair artifact paths.
- Replay state DB path.
- Event log path.
- Optional markdown and plot paths.

Output artifacts should be deterministic except for explicit timestamps that
are part of the scenario clock.

Canonical output layout:

```text
simulation/outputs/{scenario_id}/{run_id}/
  scenario.json
  results.json
  events.jsonl
  report.md
  market_data/
  artifacts/
  state/
  plots/
```

Tests may use temporary output directories instead of writing under
`simulation/outputs`.

## Safety Rules

Simulation code must fail closed:

- Live credentials are rejected.
- Live order execution is rejected unless a future integration profile
  explicitly allows it outside unit tests.
- Network calls are rejected in unit tests.
- State-only mode must not record exchange/client order ids.
- Pair recalculation must not force close positions.
- Reporting and pair-validity diagnostics must remain read-only.
- Reconciliation audit must not mutate exchange.

## Testing Strategy

Simulation itself should have tests:

- Process generation tests.
- OHLCV shape tests.
- Fault injection tests.
- Replay harness tests.
- Assertion helper tests.
- End-to-end scenario tests.

Production behavior should also be tested through simulation scenarios:

- Queue blocks future entry only.
- Existing position exits naturally.
- Missing data prevents unsafe entry.
- Stale validity produces review reasons.
- State-only run records no exchange ids.
